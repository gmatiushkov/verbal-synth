## 1 Анализ предметной области
Плавный ввод немузыкального читателя в тему и формулировка решаемой проблемы.

- **1.1 Синтез звука и задача настройки его параметров** — наполнение:
  - доступно: что такое синтезатор и «параметр» (осцилляторы, фильтр, огибающая, модуляция)
    на примере сигнальной цепи нашего синта;
  - почему ручная настройка трудна: много абстрактных параметров; неинтуитивная связь
    «число → воспринимаемый звук»; «негладкое» пространство патчей;
  - **«семантический разрыв» слова ↔ параметры вводится как явная концепция** — опорная точка работы.
  - _Вложения: Рис. 1.1 — сигнальная цепь синтезатора; Рис. 1.2 — концептуальная схема семантического разрыва (слова → ??? → 38 чисел)._

- **1.2 Обзор существующих подходов и аналогов** — наполнение (средний охват: 4 категории + таблица):
  - (а) браузеры пресетов с тегами (Serum/Vital/Massive);
  - (б) сопоставление по аудио-образцу / «обратный синтез» (academic inverse-synthesis);
  - (в) генеративные text-to-audio (MusicLM, Stable Audio) — выдают готовое аудио, а не редактируемые параметры;
  - (г) AI-генераторы патчей (например, Synplant Genopatch);
  - вывод: наша ниша — текст → **редактируемые параметры конкретного синта**, локально и быстро.
  - _Вложения: Табл. 1.1 — сравнение аналогов по критериям (вход; выход редактируемые параметры/аудио;
    привязка к синту; локальность/скорость; редактируемость). Конкретный список — уточнить мини-ресёрчем при детализации._

- **1.3 Цель и задачи работы** — наполнение:
  - цель: разработать интеллектуальную систему автоматической настройки параметров синтеза по текстовому запросу;
  - задачи (черновик): 1) анализ предметной области и аналогов; 2) проектирование и реализация
    синтезатора-платформы; 3) формализация задачи «текст→параметры» и выбор подхода; 4) сбор библиотеки
    эталонов и реализация retrieval-системы; 5) интеграция в синтезатор (Generate); 6) оценка качества.
  - _Вложения: без рисунков._

## 2 Анализ и проектирование требований
Наполнение в общих чертах (✎ согласовано 2026-06-16). Стиль 2.3 — как у образца: на каждое
средство свой под-подпункт (версия + обоснование + ссылка на источник).

- **2.1 Функциональные требования**
  - **2.1.1 Требования к синтезатору:** звук в реальном времени по MIDI/клавиатуре с полифонией;
    2 wavetable-осциллятора + шум + микшер; фильтры LP/HP (резонанс, keytrack); огибающие
    (Amp ADSR, Filter Env); модуляция (2 LFO, Filter Env → cutoff/WT/pitch/amp); эффекты (drive, reverb);
    управление 38 параметрами через GUI с визуализацией (форма волны, ADSR, АЧХ); пресеты (JSON).
  - **2.1.2 Требования к интеллектуальному модулю:** ввод короткого русского текстового запроса;
    подбор патча (38 параметров) по смыслу; применение к движку и отображение в UI (кнопка Generate);
    результат — **редактируемый** патч (можно доработать вручную).

- **2.2 Нефункциональные требования** (подпункты по категориям):
  - 2.2.1 Архитектурные — разделение «синт ↔ ML-модуль», модульность.
  - 2.2.2 Производительность и реальное время — аудио-буфер 128 сэмплов, без дропаутов на 8 голосах;
    инференс ~1–2 с, локально, **без GPU**.
  - 2.2.3 Надёжность — fallback на встроенные банки при отсутствии WAV-файлов.
  - 2.2.4 Удобство использования — мгновенный отклик ручек, понятный UI без обучения.
  - 2.2.5 Сопровождаемость и расширяемость — пополняемая библиотека прототипов.
  - 2.2.6 Платформа/совместимость — **Windows-десктоп** (целевая платформа; кроссплатформенность — потенциал JUCE).

- **2.3 Выбор средств реализации** (под-подпункты на каждое средство; версии и ссылки уточнить при детализации):
  - 2.3.1 Язык и фреймворк синтезатора — C++17 + JUCE (реальное время, DSP+GUI, индустриальный стандарт).
  - 2.3.2 Система сборки — CMake.
  - 2.3.3 Язык и библиотеки интеллектуального модуля — Python (sentence-transformers, NumPy).
  - 2.3.4 Текстовый энкодер — мультиязычный **e5-base** (обоснование выбора; эмпирический A/B — в гл. 4).
  - 2.3.5 Интеграция синтезатора и ML-модуля — запуск `predict.py` как локального подпроцесса (stdin/stdout).
  - 2.3.6 Форматы данных и хранение — JSON (пресеты, `library.json`).
  - 2.3.7 Генерация данных через LLM API (Claude) — офлайн-подготовка кандидатов для библиотеки прототипов.
  - _Примечание: при детализации проставить версии (C++17, версия JUCE, Python, версия модели) и ссылки [N]._
