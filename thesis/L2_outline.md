## ВВЕДЕНИЕ
- **Актуальность** — синтезаторы мощны, но «программирование звука» = барьер: нужно понимать
  десятки абстрактных параметров; развитие ML/NLP делает реальным интерфейс «опиши словами — получи звук».
- **Проблема** — разрыв между намерением, выраженным словами («тёплый бас», «хрустальный звон»),
  и техническими параметрами синтеза.
- **Объект** — процесс настройки параметров синтезатора; **предмет** — методы автоматической
  настройки по текстовому описанию с применением ML.
- **Цель и задачи** — кратко (развёрнуто в 1.3).
- **Краткая структура работы.**
- _Вложения: без рисунков._

## 1 Анализ предметной области
Плавный ввод немузыкального читателя в тему и формулировка решаемой проблемы.

- **1.1 Синтез звука и задача настройки его параметров:**
  - доступно: что такое синтезатор и «параметр» (осцилляторы, фильтр, огибающая, модуляция)
    на примере сигнальной цепи нашего синта;
  - почему ручная настройка трудна: много абстрактных параметров; неинтуитивная связь
    «число → воспринимаемый звук»; «негладкое» пространство патчей;
  - **«семантический разрыв» слова ↔ параметры вводится как явная концепция** — опорная точка работы.
  - _Вложения: Рис. 1.1 — сигнальная цепь синтезатора; Рис. 1.2 — концептуальная схема семантического разрыва._

- **1.2 Обзор существующих подходов и аналогов** (средний охват: 4 категории + таблица):
  - (а) браузеры пресетов с тегами (Serum/Vital/Massive);
  - (б) сопоставление по аудио-образцу / «обратный синтез» (academic inverse-synthesis);
  - (в) генеративные text-to-audio (MusicLM, Stable Audio) — выдают готовое аудио, а не редактируемые параметры;
  - (г) AI-генераторы патчей (например, Synplant Genopatch);
  - вывод: наша ниша — текст → **редактируемые параметры конкретного синта**, локально и быстро;
  - завершить **обоснованием целесообразности собственной разработки** (методичка: почему не взять готовый аналог).
  - _Вложения: Табл. 1.1 — сравнение аналогов по критериям. Список уточнить мини-ресёрчем при детализации._

- **1.3 Цель и задачи работы:**
  - цель: разработать интеллектуальную систему автоматической настройки параметров синтеза по текстовому запросу;
  - задачи (черновик): 1) анализ предметной области и аналогов; 2) проектирование и реализация
    синтезатора-платформы; 3) формализация задачи «текст→параметры» и выбор подхода; 4) сбор библиотеки
    эталонов и реализация retrieval-системы; 5) интеграция в синтезатор (Generate); 6) оценка качества.
  - _Вложения: без рисунков._

## 2 Анализ и проектирование требований
Стиль 2.3 — как у образца: на каждое средство свой под-подпункт (версия + обоснование + ссылка).

- **2.1 Функциональные требования** — уровень СПОСОБНОСТЕЙ, без привязки к реализации.
  Тип синтеза, состав осцилляторов, число параметров и т. п. — это проектные решения, они
  принимаются и обосновываются в гл. 3, а не задаются здесь.
  - **2.1.1 Требования к синтезатору:** генерировать звук в реальном времени и воспроизводить
    полифонически по нажатиям клавиш (MIDI/компьютерная клавиатура); формировать тембр из нескольких
    источников звука с их смешиванием; спектральная обработка (фильтрация) с регулируемыми параметрами;
    динамика во времени (огибающие громкости и тембра); модуляция параметров для «живого» звука;
    эффекты обработки; управление всеми параметрами через GUI с визуальной обратной связью;
    сохранение/загрузка конфигураций (пресетов).
  - **2.1.2 Требования к интеллектуальному модулю:** ввод короткого русского текстового запроса;
    формирование полного набора параметров синтезатора («патча») по смыслу запроса; применение
    результата к синтезатору и отображение в UI (кнопка Generate); результат — **редактируемый**
    патч (можно доработать вручную).

- **2.2 Нефункциональные требования** (подпункты по категориям):
  - 2.2.1 Архитектурные — разделение «синт ↔ ML-модуль», модульность.
  - 2.2.2 Производительность и реальное время — аудио-буфер 128 сэмплов, без дропаутов на 8 голосах;
    инференс ~1–2 с, локально, **без GPU**.
  - 2.2.3 Надёжность — fallback на встроенные банки при отсутствии WAV-файлов.
  - 2.2.4 Удобство использования — мгновенный отклик ручек, понятный UI без обучения.
  - 2.2.5 Сопровождаемость и расширяемость — пополняемая библиотека прототипов.
  - 2.2.6 Платформа/совместимость — **Windows-десктоп** (целевая; кроссплатформенность — потенциал JUCE).

- **2.3 Выбор средств реализации** (под-подпункты; версии и ссылки уточнить при детализации):
  - 2.3.1 Язык и фреймворк синтезатора — C++17 + JUCE (реальное время, DSP+GUI, индустриальный стандарт).
  - 2.3.2 Система сборки — CMake.
  - 2.3.3 Язык и библиотеки интеллектуального модуля — Python (sentence-transformers, NumPy).
  - 2.3.4 Текстовый энкодер — мультиязычный **e5-base** (обоснование выбора; эмпирический A/B — в гл. 4).
  - 2.3.5 Интеграция синтезатора и ML-модуля — запуск `predict.py` как локального подпроцесса (stdin/stdout).
  - 2.3.6 Форматы данных и хранение — JSON (пресеты, `library.json`).
  - 2.3.7 Генерация данных через LLM API (Claude) — офлайн-подготовка кандидатов для библиотеки прототипов.
  - _Примечание: проставить версии (C++17, JUCE, Python, модель) и ссылки [N] при детализации._

## 3 Разработка программы
Ядро записки. Сначала общая картина, затем инструмент, затем интеллект над ним.

- **3.1 Проектирование общей архитектуры системы:** две части — синтезатор (C++/JUCE, реальное время)
  и интеллектуальный модуль (Python); связь — Generate → `predict.py` (подпроцесс) → 38 параметров → `applyPatch`.
  - _Вложения: Рис. 3.1 — общая архитектура системы._

- **3.2 Синтезатор (звуковой движок)** ← здесь ПРИНИМАЕМ и ОБОСНОВЫВАЕМ конкретные решения
  - 3.2.1 Архитектура движка: **выбор и обоснование wavetable-синтеза**; конкретная конфигурация —
    2 wavetable-осциллятора + шум + микшер, 6 банков, фильтры LP/HP, огибающие, 2 LFO, эффекты;
    сигнальная цепь, 8 голосов, wavetable-осциллятор (16×2048, билинейная интерполяция).
    - _Вложения: Рис. 3.2 — сигнальная цепь голоса; Рис. 3.3 — схема модуляции (LFO/env routing)._
  - 3.2.2 Модель параметров: состав и число параметров как СЛЕДСТВИЕ архитектуры (**38** нормализованных
    [0,1]); дискретные vs непрерывные; конвертер norm↔real зеркалит движок.
    - _Вложения: Табл. 3.1 — параметры (возможно в приложение)._
  - 3.2.3 UI и MIDI: PatchPanel, ручки/фейдеры, визуализаторы (волна/ADSR/АЧХ), анимация модуляции,
    клавиатура, пресеты.
    - _Вложения: Рис. 3.4 — макет UI._

- **3.3 Интеллектуальная система автоматической настройки параметров** ← фокус темы ВКР
  - 3.3.1 Постановка задачи: текст → 38 параметров; два типа информации в запросе — идентичность (A, «что за
    инструмент») и модификаторы (B, «какой именно»); почему задача трудна (мало данных, негладкое пространство).
  - 3.3.2 Выбор подхода и обоснование: кандидаты (прямая регрессия всех параметров / классификация по параметрам /
    генерация «текст→аудио» / подбор эталона-прототипа из библиотеки) → выбран **подбор прототипа обучаемой
    моделью + параметрические модификаторы**; почему (нет схлопывания к среднему, сохраняется специфика звука,
    результат редактируемый, качество = библиотека эталонов + обучаемая голова, а не размер модели; локальный запуск).
  - 3.3.3 Данные: библиотека прототипов (140 эталонов; `library.json`: 38 параметров + текстовые дескрипторы);
    как собрана (кандидаты через LLM-API + ручной тюнинг на слух); обучающий набор формулировок
    (`queries.json`: ~30 перефразировок на прототип, ≈4275 пар «формулировка → прототип»); схема именования.
  - 3.3.4 Метод и архитектура (**каскад L1→L4**):
    - L1 — представление запроса: предобученный текстовый энкодер **e5-base** (заморожен как извлекатель
      признаков) → числовой вектор (эмбеддинг);
    - L2 (**обучаемое ядро**) — классификатор: небольшая обучаемая «голова» (полносвязная сеть, MLP) над
      замороженным эмбеддингом → распределение вероятностей по 140 прототипам → выбор прототипа (идентичность);
      обучается на ~4275 парах «формулировка → прототип»; **почему обучаемая голова, а не сырой поиск ближайшего
      по косинусу (retrieval):** голова учит границы решений на множестве формулировок, а не полагается на
      неадаптированную близость в пространстве энкодера (это чинит ошибки сырого поиска на «голых» существительных);
    - L3 — параметры прототипа: выбранному прототипу соответствует готовый ручной набор 38 параметров (`library.json`);
    - L4 — параметрические модификаторы: слова-модификаторы запроса (теплее/ярче/короче/ниже/агрессивнее/просторнее)
      детерминированно сдвигают ТОЛЬКО затронутые непрерывные параметры в перцептивной (логарифмической) шкале,
      не ломая ядро прототипа; учитываются усилители («очень/чуть») и отрицание («не яркий»);
    - сырой retrieval (косинусная близость) сохраняется как **baseline** для сравнения (гл. 4);
    - теория эмбеддингов и обучаемого классификатора вплетается по месту.
    - _Вложения: Рис. 3.5 — каскад L1→L4 (запрос → энкодер e5 → классификатор → прототип → модификаторы → 38 чисел)._
  - 3.3.5 Интеграция: Generate → `predict.py` (stdin/stdout), на лету без перебилда C++; чистый stdout / диагностика stderr.

## 4 Тестирование и оценка
Оценка качества интеллектуальной системы; опора на реальные артефакты автоэвала
(`eval_*.py`, `train_clf.py` с отложенной выборкой). Конкретные числа — свести
с актуальным кодом/прогонами при углублении.

- **4.1 Методика оценки:**
  - почему НЕ MSE по параметрам (негладкое мультимодальное пространство → схлопывание к среднему);
    оцениваем по ИДЕНТИЧНОСТИ (правильно ли выбран прототип) и по АУДИО;
  - две выборки: отложенная тестовая (стратифицированный сплит формулировок, обучаемая модель её не видит)
    и независимый golden-набор «запрос → ожидаемый прототип»; метрики **top-1 / top-3** (strict и по роли);
  - аудио-санити: рендер патча → мел-спектр → косинус к эталону;
  - оценка модификаторов: направленность сдвигов («ярче» → spectral centroid ↑, «короче» → огибающая короче),
    срабатывание отрицания/усилителей;
  - автоматизация прогонов (скрипты `eval_*.py`).

- **4.2 Экспериментальное сравнение:**
  - **главный результат — обучаемая голова (классификатор) vs сырой поиск ближайшего (retrieval-baseline):**
    top-1/top-3 на тесте и golden (по роли TEST 61→82 %, GOLDEN 93→96 % — голова чинит провалы baseline);
  - A/B текстовых энкодеров для слоя представления — **e5-base vs MiniLM** (числа + вывод, почему e5-base);
  - синтетический эвал.
  - _Вложения: Табл. 4.1 — метрики (классификатор vs retrieval-baseline, top-1/top-3); Рис. 4.1 — A/B энкодеров (matplotlib)._

- **4.3 Тестирование пользовательских сценариев:**
  - качественные кейсы «запрос → результат» (напр.: «тёплый бас», «хрустальный звон», «acid bass», «вибрафон»);
  - демонстрация модификаторов («ярче» / «короче» / «не яркий»);
  - известные ограничения (запросы из одних модификаторов без инструмента; гибриды «гитарный пэд» — слабее).
  - _Вложения: Табл. 4.2 — примеры «запрос → подобранный прототип/патч»._

- **4.4 Выводы по результатам:** что подтвердилось (идентичность ловится; e5-base лучше MiniLM),
  ограничения, направления улучшения.
