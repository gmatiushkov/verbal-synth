# VerbalSynth — ПИВОТ: «текст → патч» через Retrieval + модификаторы

> **Статус:** план-of-record (2026-06-15). **Отменяет** генеративный подход v3/v4/v5
> (`DATASET_v5_PLAN.md`) как ОСНОВНОЙ. Возник после провала ML-генерации параметров и
> исследования (см. `RESEARCH_BRIEF.md` + два независимых анализа LLM). Этого файла + ссылок
> достаточно, чтобы войти в работу из новой сессии.

---

## 0. TL;DR

Задача: русский запрос («тёплый бас», «acid bass», «вибрафон», «хрустальный звон») → 38 нормализованных
параметров патча. Кнопка Generate в синте (C++/JUCE) зовёт локальный Python (`predict.py`).

**Решение (консенсус 3 анализов + наши замеры): НЕ генерировать патч, а ВЫБИРАТЬ ближайший
эталон из библиотеки + модулировать.**
```
запрос → энкодер → RETRIEVAL ближайшего эталона (идентичность, дискретные параметры) 
        → [фаза 2] ML-модификатор (непрерывные оси: яркость/атака/тело/...) → 38 параметров
```
- **Идентичность не генерируется, а выбирается** → нет «схлопывания к среднему», специфика сохраняется
  (вибрафон ≠ маримба, т.к. это РАЗНЫЕ прототипы в библиотеке, а не одна роль).
- **Дискретные параметры (банк/октава/форма LFO) берутся ИЗ прототипа** (retrieval), ML их не трогает.
- **Качество = библиотека ручных эталонов** (главный актив), а не мощность модели.

## 1. Почему так (диагноз)

- Прямая регрессия 38 чисел → схлопывание к среднему (оптимум MSE на мультимодальных данных). Замерено.
- Пространство патчей НЕ гладкое: близкие по числам звучат по-разному и наоборот → «текст→38 чисел»
  плохое представление.
- Декомпозиция в ~31 «роль» теряла специфику (вибрафон/маримба в одной роли). Отвергнуто.
- Синтетические данные из одной спеки учат «стиль Claude / восстановление спеки», не намерение юзера.
- Замер на golden (реалистичные промпты): банк понимается (~80–100%), но непрерывные оси плыли;
  пробинг энкодера — сигнал есть, но слабый (роль +28 п.п., оси +10–20). Подробности — в памяти/`DATASET_v5_PLAN.md`.

## 2. Архитектура (фазами; сроки сжаты)

### MVP (фаза 1) — RETRIEVAL ONLY
`запрос → энкодер → косинус к описаниям прототипов → top-1 прототип → отдаём его 38 параметров как есть.`
- Всегда звучит хорошо (это ручной эталон). Проверяет понимание ИДЕНТИЧНОСТИ.
- ML = эмбеддинги + поиск (FAISS/косинус). Без обучения отдельной модели на первом шаге.
- Вариативность пока = разные прототипы под разные запросы.

### Фаза 2 — + МОДИФИКАТОРЫ (после MVP)
Лёгкая модель: `запрос → уровни осей (яркость/атака/тело/движение/пространство/толщина)` →
детерминированно сдвигаем НЕПРЕРЫВНЫЕ параметры прототипа (с ограниченным диапазоном, чтобы не
сломать ядро архетипа). Дискретные не трогаем. Переиспользует наш слой осей (`attributes.json`/`attr_decode.py`).

### НЕ делаем (при наших данных): CVAE, диффузия, flow, прямая регрессия 38, большой генератор.

## 3. Библиотека прототипов (КРИТПУТЬ)

**Узкое место = не модель, а число качественных эталонов.** Сейчас ~40 (28 `[REF]` + 12 заводских),
цель ~100–150 (покрыть таксономию ~140 целей). Расширение: **Claude рисует кандидата → юзер слушает/
правит/утверждает в синте** (стриминг: патчи появляются в синте пачками, тюнятся по ходу).

### Система именования (для синта; сортируется по имени файла)
`NNN <русское имя цели>.json`, NNN — глобальный 3-значный номер в порядке таксономии (банки/категории
идут блоками: акустика 001–044, синт-роли 045–078, текстуры 079–102, перкуссия/эффекты 103–126,
барабаны 127–140). 1 прототип = 1 цель таксономии (MVP). Все существующие патчи приведены к этой схеме.
- Источник правды — `ml/data/library/library.json` (см. ниже). Файлы в `Patches/` = презентация для
  тюнинга. Юзер правит в синте → пересохраняет под тем же именем → `sync` тянет params обратно в library.

### library.json (источник правды)
Список прототипов: `{num, name, target, category, role, bank, source(ref|factory|gen), approved,
params(38 norm), descriptors[], attributes{register/brightness/body/bank...}}`.
- `params` — параметры эталона. `approved` — утверждён ли юзером на слух.
- `descriptors`/`attributes` — СЕМАНТИКА для генерации запросов (см. §4), НЕ выводятся из 38 чисел.

## 4. Тексты для retrieval (как Claude делает описания)

Для retrieval нужно сопоставлять запрос с описаниями прототипов. Генерим НЕ патчи, а **много
пользовательских ФОРМУЛИРОВОК на каждый прототип** (20–50): синонимы, сленг, жанр, эмоция
(«acid bass» → кислотный бас / tb303 / резонансный бас / рейв-бас / squelch …).
- **На чём Claude основывает запросы:** на СЕМАНТИКЕ прототипа из library.json — `target` (имя),
  `descriptors` (из таксономии), `attributes` (регистр/яркость/тело/банк), НЕ на 38 числах.
- Эти тексты → индекс retrieval (эмбеддинги). Реальные запросы юзера матчатся к ближайшему.
- ВАЖНО (риск train/real mismatch): по возможности добавить РЕАЛЬНЫЕ формулировки, не только Claude.

## 5. Оценка (только по аудио, не MSE параметров)

- **Identity (главное):** golden-набор `запрос → ожидаемый прототип`; метрика retrieval **top-1/top-3**.
  (`ml/eval/golden_prompts.json` переразметить на «ожидаемый прототип/цель».)
- **Аудио-санити:** рендер патча → мел-спектр → косинус к эталону.
- **Модификаторы (фаза 2):** направленность — «ярче» → spectral centroid выше; «короче» → огибающая короче.
- Энкодер: дешёвый A/B (MiniLM vs e5/bge-m3/LaBSE/эмбеддинги API) по top-1 на golden, без обучения.

## 6. Что переиспользуем / выкидываем

**Переиспользуем:** ручные эталоны → библиотека; `gen_reference.py`/`system_prompt_reference.md` →
генерация кандидатов; `param_convert.py` → рендер/аудио-метрики; `attributes.json`/`attr_decode.py` →
модификаторы; `golden_prompts.json` → identity-эвал; `probe_encoder.py` → A/B энкодера; пайплайн
описаний → синоним-запросы.
**Откладываем/выкидываем:** «генерировать 38 чисел», роль-коллапс, 560-патчевый датасет как обучающую
цель, регресс/классиф-модель (`ml/models/v5`,`v5b` — архив), train.py-голова.

## 7. План работ (порядок)

1. ✅ Документ (этот файл).
2. **Библиотека+именование:** `build_library.py` → library.json + нумерованные пресеты для покрытых
   (~40) + список пробелов (~100); чистка Patches от старых имён/`[V5]`.
3. **Стриминговая генерация пробелов:** `gen_library.py` (Claude, по 1 цели, чекпойнт+импорт в синт сразу) →
   юзер тюнит по ходу; `sync_library.py` тянет правки обратно.
4. **Retrieval MVP:** энкодер A/B → индекс по синоним-запросам прототипов → `predict.py` отдаёт top-1 прототип.
5. **Фаза 2:** модификаторы осей.

## 8. Инвентарь файлов

- `ml/data/library/library.json` — источник правды (прототипы).
- `ml/scripts/build_library.py` / `gen_library.py` / `sync_library.py` — сборка/генерация/синк библиотеки.
- `ml/config/system_prompt_reference.md` — промпт генерации патча-кандидата (params, реальные единицы).
- `ml/config/v5/taxonomy_to_role.json` — 140 целей (порядок нумерации), descriptors, axis_preset.
- `ml/data/reference/reference_anchors.json` — 28 утверждённых `[REF]` (источник params для них).
- `ml/scripts/param_convert.py` — norm↔real (зеркало движка), рендер-истина.
- `ml/eval/golden_prompts.json` — эвал идентичности (переразметить на «ожидаемый прототип»).
- `Source/MainComponent.cpp` — кнопка Generate (зовёт predict.py); пресеты грузятся по имени параметра.
- Движок/единицы — `DATASET_v5_PLAN.md` §10 (факты не переоткрывать).
