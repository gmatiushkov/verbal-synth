## Пересмотр v2 → v3 — Реальные единицы (АКТУАЛЬНЫЙ ПЛАН)

### Проблема, вскрытая на тест-выборке

Патчи v2 систематически ошибались в огибающих (ADSR/Filter Env), частотах LFO, drive — несмотря
на то, что в промпт были добавлены подробные подсказки вида «0.3 ≈ 8 мс, 0.5 ≈ 50 мс».
**Корень:** все параметры синтезатора нормализованы в `[0,1]` через **логарифмическую** формулу
`SynthUtils::logParam: real = min · (max/min)^norm`. Claude вынужден держать в уме этот логарифм
для каждого времени/частоты — и ошибается. В реальных единицах (мс, Гц, центы) он рассуждает
естественно и точно, как живой саунд-дизайнер.

### Ключевая идея: перевернуть направление конвертации

| | Было (v2) | Стало (v3) |
|---|-----------|-----------|
| Что выдаёт Claude | нормализованные `[0,1]` | **реальные величины** (мс, Гц, центы, Q, %, имена банков) |
| Конвертация в `[0,1]` | делает сам Claude (в уме, логарифм) | детерминированный Python-конвертер `param_convert.py` |
| Нормализованные `[0,1]` | вход и выход | только **выход** для обучения; Claude их не видит |
| Источник формул | проза в param_reference (частично неверная) | **C++-движок** (единственная истина), скопирован построчно |

Нормализованные `[0,1]` по-прежнему нужны (вход синтезатора + обучающий вектор) — но
теперь они вычисляются точной формулой из реальных значений, а не «на глаз» Claude.

### Философия промпта (важно — чтобы не перегрузить Claude)

Промпт даёт **факты, а не рецепты**:
- **Объективная механика синта** (единицы, диапазоны, кривые, что физически делает каждый
  параметр) — это истина движка, она ДОЛЖНА присутствовать и быть **точной**.
- Claude **уже владеет** общим саунд-дизайном (знает, что у смычка мягкая атака, у вибрато
  частота ~5–6 Гц, у acid — резонансный свип фильтра). Это **не нужно** диктовать рецептами по
  каждому инструменту — это переобучает на узкие случаи и перегружает контекст.
- Поэтому: точные единицы + точные роли параметров + **немного общих принципов** (форма
  огибающей определяет тип звука; что с чем связано) + **few-shot из реальных заводских пресетов**.
  Дальше — доверяем экспертизе Claude.
- **Конкретные числа в промпте допустимы только если это** (а) истина движка (диапазон/кривая
  параметра) или (б) общеизвестный физический факт, проверенный. **Никаких выдуманных значений**
  и минимум прескриптивных «звук X = параметры Y».
- Реакция на фидбек по конкретным патчам — не «вписать рецепт этого звука в промпт», а понять,
  какого общего знания о механике синта не хватило, и дать именно его.

### Реальные единицы (извлечены из C++ — единственная истина)

Формулы из [Source/SynthUtils.h](Source/SynthUtils.h), [Source/SynthVoice.cpp](Source/SynthVoice.cpp),
[Source/SynthFilter.cpp](Source/SynthFilter.cpp):

| Параметр(ы) | Формула `real ← norm` (C++) | Единица / диапазон |
|---|---|---|
| lp_cutoff, hp_cutoff | `logParam(n, 20, 18000)` | Гц, 20…18000 (лог) |
| amp_attack, fenv_attack | `logParam(n, 0.0005, 5)` | с, 0.0005…5 (лог) |
| amp_decay, fenv_decay | `logParam(n, 0.001, 5)` | с, 0.001…5 (лог) |
| amp_release, fenv_release | `logParam(n, 0.005, 10)` | с, 0.005…10 (лог) |
| lfo1_rate, lfo2_rate | `logParam(n, 0.01, 20)` | Гц, 0.01…20 (лог) |
| lp_resonance, hp_resonance | `q = 0.5 + n·9.5` | Q, 0.5…10 (лин) |
| osc1_octave, osc2_octave | `round(n·4) − 2` | окт, −2…+2 (дискр., шаг 1) |
| osc2_detune | `(n − 0.5)·100` | центы, −50…+50 (лин) |
| osc2_semitones | `round((n − 0.5)·48)` | полутоны, −24…+24 (дискр.) |
| fenv_amount | `(n − 0.5)·2`, затем ×4 окт свип | окт свипа, −4…+4 (0=нейтр) |
| lfo1_to_pitch, lfo2_to_pitch | глубина ×2 полутона | центы, 0…200 |
| lfo1_to_filter | глубина ×2 октавы | окт, 0…2 |
| osc*_table | snap {0,.2,.4,.6,.8,1.0} | имя банка (Basic…Digital) |
| lfo*_shape | snap {0,.33,.67,1.0} | имя (sine/triangle/saw/square) |
| osc*_position | морф 0…1 | % 0…100 (+ position_map банка) |
| mix_*, *_sustain, drive_*, reverb_*, fenv_to_wt, lfo*_to_wt, lfo*_to_amp, filter_keytrack | линейные | **% 0…100** |

Решения по представлению (согласовано):
- Линейные уровни (mix/sustain/drive/reverb/keytrack/mod-depths) → Claude видит **проценты 0–100%**.
- Дискретные банки/формы LFO → **имена строками**, не числа.
- В `param_reference.json` v3 хранятся реальные диапазоны+кривые; нормализованная проза удаляется.

### Что переделать (по файлам)

1. **🆕 `ml/scripts/param_convert.py`** — модуль конвертации real↔norm, **зеркало C++**
   (каждая формула с комментарием-ссылкой на строку исходника). Round-trip самотест
   `norm → real → norm ≈ identity` на сетке значений — защита от расхождения с движком.
   *Пишется и тестируется ПЕРВЫМ — на нём держится корректность всех патчей.*
2. **♻️ `param_reference.json` → v3** — двунаправленная спека: для каждого из 38 параметров
   `unit`, `real_min`/`real_max`, `curve`, имена дискретных. Нормализованная проза удаляется.
3. **♻️ `system_prompt.md` → v3** — таблица параметров в реальных единицах + точные перцептивные
   ориентиры (только истина движка / проверенные факты) + общие принципы огибающих + **2–3 few-shot
   из заводских пресетов** (Piano/Violin/Organ — разнотипные, с краткой логикой «почему так»).
   Удалить нормализованную математику и переобучающие рецепты по конкретным звукам.
4. **♻️ `generate_dataset.py`** — Claude → JSON в реальных единицах → `param_convert` → norm.
   В `patches_raw.json` хранить **оба** представления (real — для ревью глазами, norm — для экспорта).
   **structured outputs убран** (нестабилен у прокси) → JSON через требование в промпте + парсер.
   Мерж/запись только средствами Python (UTF-8 без BOM).
5. **♻️ `validate_patches.py`** — валидация и флаги качества в **реальных единицах** (читаемо:
   «attack 8 мс при описании „мягкий смычок"»), затем конверт в norm для синтезатора и `dataset.jsonl`.
6. **❌ C++ не трогаем** — остаётся потребителем `[0,1]`. Формат `dataset.jsonl` (вектор 38 norm)
   не меняется → обучение не затрагивается.
7. **🗑️ Убрать лишнее:** нормализованные вейпоинт-таблицы; `archetypes.json` (deprecated) — ✅ удалён.

### Главный риск и контроль

Конвертер real→norm обязан быть **точным обратным** к движку. Если формулы разойдутся — патч
зазвучит не так, как Claude задумал. Защита: round-trip тест в `param_convert.py` + проверка
нескольких known-точек против C++.

### Порядок работ

1. `param_convert.py` + round-trip тест ✅ → 2. `param_reference.json` v3 →
3. `system_prompt.md` v3 + few-shot → 4. адаптация `generate_dataset.py` / `validate_patches.py` →
5. `--dry-run` (бесплатно, ревью промпта) → 6. тест 4–6 патчей → валидатор → **прослушивание**.

### Заметка по кодировке (разобрано)

«Битые» описания в JSON оказались **не** багом генератора: файл — корректный UTF-8. Виноваты были
(а) вывод консоли PowerShell в cp1251 и (б) BOM, добавленный мержем через `Out-File`. Лечится тем,
что чтение — `utf-8-sig` (уже внесено), а запись/мерж — только в Python (без BOM).

### Прокси aiprimetech и стоимость — разобрано на тесте (2026-06-14)

- **Корень «пустых ответов»:** прокси инжектит агентные инструменты, и Opus периодически отдаёт
  `stop_reason=tool_use` (фиктивный вызов WebFetch) вместо текста. **Фикс:** явный запрет в промпте
  («не вызывай инструменты/функции, ответ — только текст с JSON») → 6/6 чисто против ~50% и падения
  целых категорий в 0. Дополнительно `_one_call` переключён на **create-primary** (стрим на tool_use
  отдаёт пустой SSE), стрим — запасной. Ретраи идут **без thinking** (провал стоит копейки).
- **Кэш НЕ работает** на этом прокси (`cache_read=0` на идентичном ~23.8K-префиксе) ни на Default,
  ни на Max[1m] → тариф **Default (0.7×)** верный, Max = чистый штраф ×1.3. Модель — `claude-opus-4-8`
  (без `[1m]`). Системный промпт срезан ~25K→~23K ток (за счёт wavetable-summaries; правила нетронуты).
- **Реальная стоимость** (без кэша, чистый прогон, доминирует выход/thinking): **~$24–28** на ~800
  патчей при батче 6–8. Старая оценка ($7–15) была завышенно-оптимистична (предполагала рабочий кэш).

## 🚀 ЗАПУСК ПОЛНОГО ДАТАСЕТА — рабочий runbook (актуально 2026-06-14, v3 финал)

> Этот раздел — **единственный источник правды по запуску**. Старые блоки ниже («Оценка стоимости v2»,
> «Целевой объём v2», «Решения v2») оставлены как история; где расходятся — верен этот раздел.

### Что мы делаем
Генерируем датасет **(русский текст → 38 нормализованных float)** для обучения модели MiniLM+MLP,
которая встанет за кнопку **Generate** в синтезаторе (текст → патч). Claude проектирует каждый патч в
**реальных единицах** (Гц/мс/центы/%) + 6 русских описаний; `param_convert.py` детерминированно
переводит в `[0,1]`. Разнообразие держится на: **139 целей в 5 категориях** + **каждый запрос = N
разных вариантов ОДНОЙ цели** (мандат «не-близнецы, ≥3 оси» бьёт по спреду повторов одной цели).

### Файлы и роли
| Файл | Роль |
|---|---|
| `ml/config/sound_taxonomy.json` | 139 целей, 5 категорий (acoustic/synth/texture/percussive/drums), доли |
| `ml/config/system_prompt.md` | шаблон промпта: факты движка + few-shot (12 пресетов) + правила описаний |
| `ml/config/param_reference.json`, `wavetable_doc.json` | данные для подстановки в промпт |
| `ml/config/api_config.local.json` | креды: base_url=`https://aiprimetech.io`, model=`claude-opus-4-8`. **НЕ коммитить** |
| `ml/scripts/param_convert.py` | конвертер real↔norm (зеркало C++), round-trip самотест |
| `ml/scripts/generate_dataset.py` | генерация (Claude→JSON real→norm), чекпойнт после каждого батча |
| `ml/scripts/validate_patches.py` | кламп/снап + флаги качества + экспорт + разворот в `dataset.jsonl` |
| `ml/data/patches_raw.json` | **АККУМУЛЯТОР** всех патчей: `params`(norm)+`params_real`+`target`+6 описаний |
| `ml/data/dataset.jsonl` | финальные обучающие строки (1 на описание = патчей×6) |

### Команды запуска
Креды берутся из `api_config.local.json` автоматически. Запускать из `ml/scripts/`,
с `PYTHONIOENCODING=utf-8` (иначе кириллица в консоли крашит print).

**Вариант А — всё разом (проще; чекпойнт после каждого батча):**
```
python generate_dataset.py --all --count 1200 --batch-size 8
```
Пишет в `ml/data/patches_raw.json` (там уже 30 сид-патчей). При обрыве — повтор той же команды
дособерёт по-целевой дефицит (готовые цели пропускаются).

**Вариант Б — по частям (по категориям; удобнее мониторить и проверять покатегорийно):**
```
python generate_dataset.py --category acoustic_emulation --count 336 --batch-size 8 --out ../data/patches_raw.json
python generate_dataset.py --category synth_roles        --count 312 --batch-size 8 --out ../data/patches_raw.json
python generate_dataset.py --category texture_atmos      --count 216 --batch-size 8 --out ../data/patches_raw.json
python generate_dataset.py --category percussive_fx      --count 192 --batch-size 8 --out ../data/patches_raw.json
python generate_dataset.py --category drums              --count 144 --batch-size 8 --out ../data/patches_raw.json
```
(числа — для total=1200; для 1000: 280/260/180/160/120. **Обязательно `--out ../data/patches_raw.json`** —
иначе категория уйдёт в отдельный файл `patches_<cat>.json`.)

### Что сохраняется куда / чекпойнт / резюме
- `patches_raw.json` перезаписывается **атомарно после КАЖДОГО батча** (tmp+replace) → обрыв не теряет прогресс.
- Резюме: повтор той же команды — `generate_category` считает уже сделанное **по каждой цели** (поле `target`)
  и добивает только дефицит. Полностью набранные цели/категории пропускаются.
- Никакого «всё заново» не бывает — это и есть встроенное дробление (на уровне батча/цели).

### Мониторинг прогресса
Вывод по ходу: заголовок категории → по каждой цели `[цель] +N → got/want (in/out токены)` →
после категории `══► ПРОГРЕСС done/total (%) · прошло N мин`. Проверить со стороны в любой момент:
```
python -c "import json;d=json.load(open('../data/patches_raw.json',encoding='utf-8-sig'));print(len(d['entries']),'патчей')"
```

### Время и стоимость (Default, батч 8, последовательно)
- **~3.5–4 часа** на 1000–1200 патчей (≈18 c/патч с учётом thinking и latency). По категориям — ~40–60 мин каждая.
- **~$24–36** (1000→$30, 1200→$36). Доминирует ВЫХОД (thinking). Кэша нет (учтено).
- Ускорить можно параллелью запросов, но на этом нестабильном прокси **не рекомендуется** — последовательно + чекпойнт надёжнее.

### Что может пойти не так → защита
| Риск | Защита |
|---|---|
| Прокси отдаёт пустое/`tool_use` | анти-инструмент-правило в промпте + create-primary + дешёвые ретраи без thinking; при полном провале цели — добор на повторном запуске |
| Обрыв/таймаут/перезагрузка | чекпойнт после каждого батча → повтор команды резюмит |
| Невалидный патч (<30 параметров / битый JSON) | пропускается, дефицит добирается на повторе |
| Усечение по `max_tokens` (24000) | для батча 8 есть запас; при усечении JSON → ретрай |
| Перерасход бюджета | объём фиксирован `--count`; следить по токенам в выводе |

### После генерации (Фаза 2 → 3)
1. `python validate_patches.py --in ../data/patches_raw.json --jsonl ../data/dataset.jsonl` →
   кламп/снап + флаги качества + `dataset.jsonl` (патчей×6 строк) + (опц.) экспорт плоских JSON для прослушки.
2. Прослушать 8–10 случайных патчей на категорию; просмотреть флаги (вибрато/атака/тишина/клип).
3. **Бэкап `patches_raw.json`** — ценный артефакт (часы генерации + деньги). Закоммитить или сохранить копию.
4. Фаза 3 ([ML_IMPLEMENTATION.md](ML_IMPLEMENTATION.md)): `train.py` (MiniLM-L6 заморожен + MLP→38 sigmoid,
   train/val сплит, HuberLoss) → `export_onnx.py` → интеграция в `MainComponent.cpp` (кнопка Generate).

### Что держать в голове
- Тариф **Default** + модель **`claude-opus-4-8`** (без `[1m]`). Кэш на прокси не работает — это нормально, учтено.
- Промпт = **факты, не рецепты**. Лимит движка: **нет pitch-envelope** (one-shot питч-дроп невозможен).
- 30 divtest-патчей уже в `patches_raw.json` (сид). Из них 3 (808) в описаниях упоминают «питч-дроп» —
  мелкий шум на ~6000 строк; при желании можно выкинуть перед обучением.

## Главное изменение v1 → v2

**Было (v1):** скрипт случайно генерирует патчи в рамках диапазонов архетипа → Claude по сухим числам пишет описания.

**Стало (v2):** **Claude сразу проектирует конкретный звук** — придумывает задумку, выставляет 38 параметров под неё и пишет 6 описаний. Одним запросом он отдаёт патч + описания.

**Почему так лучше:**
- Случайные патчи в диапазонах получаются однообразными и безликими; по сухим числам Claude трудно придумать осмысленное описание.
- Когда Claude сам проектирует звук под идею («виолончель в низком регистре»), патч получается *интенциональным и связным*, а описание гарантированно ему соответствует.
- Разнообразие достигается прямым мандатом на вариативность, а не нарезкой случайных чисел.

**Следствие:** система архетипов с жёсткими диапазонами параметров **упраздняется**. Вместо неё —
[sound_taxonomy.json](ml/config/sound_taxonomy.json): карта пространства звуков (категории + примеры целей + оси разнообразия) **без** диапазонов параметров.

**Что нужно дать Claude, чтобы он «слышал» синтезатор:**
1. Полную таблицу 38 параметров с маппингом `0..1 → реальные величины` — [param_reference.json](ml/config/param_reference.json) (учитывает лог-кривые: cutoff 20–18000 Гц, времена 0.5мс–10с и т.д.).
2. Перцептивную документацию банков (`perceptual_summary` + `position_map`) — уже готова в [wavetable_doc.json](ml/config/wavetable_doc.json) (Фаза 0).
3. Эвристики саунд-дизайна (как получить плак/пэд/бас/колокол/вибрато…).
4. Мандат на разнообразие + спектр целевых звуков (имитации инструментов, характеры).
5. Правила 6 стилей описаний + словарь дескрипторов.

Всё это собрано в [system_prompt.md](ml/config/system_prompt.md).

## Конфигурация API

Прокси aiprimetech (Anthropic-совместимый SDK).

```python
import anthropic
client = anthropic.Anthropic(
    api_key="YOUR_AIPRIMETECH_KEY",     # вставить позже
    base_url="https://api.aiprimetech.io/v1",
)
MODEL = "claude-opus-4-8"
```

### Выбор тарифа (бюджет $25)

| Тариф | Коэф. | Контекст | Вывод |
|-------|------|----------|-------|
| **Default** ✅ | 0.7× | 200K | Оптимум цена/мощность |
| Max | 1.3× | 1M, 99.6% cache hit | Только если нужен 1M-контекст |

Берём **Default**: системный промпт ~10–14K токенов (param-таблица + банки + правила) свободно влезает в 200K; платить 1.3× за 1M-контекст незачем. Default почти вдвое дешевле Max.

### Как делать запросы

- **System-промпт = стабильный префикс** (синт-доки + банки + словари + схема) → один `cache_control: {type:"ephemeral"}` на последнем system-блоке. Читается из кэша во всех запросах (~0.1× стоимости).
- **User-сообщение = переменная часть:** «Сгенерируй N патчей категории X, цели: [...], варьируй по осям [...]».
- **structured outputs** (`output_config.format` с json_schema) — гарантирует парсимый JSON. Opus 4.8 поддерживает.
- **Adaptive thinking** включён (лучше проектирует), `effort:"medium"` для экономии; батчинг 6–8 патчей/запрос амортизирует «размышления».
- **Batch API** (`/v1/messages/batches`, −50%) для массовой генерации — не латентно-критично. Если прокс не поддерживает batch — работаем в обычном режиме (бюджет всё равно проходит).
- **Чекпойнты:** дописывать результат в JSONL построчно (append-only), уметь продолжить с места обрыва.

### Оценка стоимости (Opus 4.8, Default 0.7×)

Допущения: ~1000 выходных токенов на патч (вкл. долю thinking + JSON + 6 RU-описаний); система кэшируется.

| Сценарий | Патчей | Out-токены | Стоимость вывода | + Batch −50% |
|----------|--------|-----------|------------------|--------------|
| Тест | ~60 | ~60K | ~$1.1 | ~$0.5 |
| Полный v1 | ~800 | ~800K | ~$14 | ~$7 |

Вход/кэш-чтение системного промпта ≈ $0.5 суммарно (незначимо). Итог: **Opus Default + Batch ≈ $7–8** на полный датасет → укладывается в $25 с запасом на итерации/перегенерацию. Без Batch ≈ $14–15 — тоже проходит. Если нужно экономить ещё — часть категорий генерировать Sonnet 4.6.

## Фаза 1 — Тестовая выборка (проверка качества)

**Цель:** убедиться, что патчи звучат как заявлено и описания адекватны — по **всем типам**, а не 4 архетипам.

**Объём:** ~3–5 патчей × каждая из 4 категорий = ~16–20 патчей × 6 = ~120 примеров.
Внутри каждой категории просить разные цели (например для acoustic_emulation: гитара, виолончель, флейта, орган, колокол) — чтобы проверить разные банки/приёмы.

### Шаги
1. Написать `generate_dataset.py` (сборка system-промпта из конфигов; structured outputs; чекпойнты).
2. Запустить в режиме `--test`: по N патчей на категорию, цели берутся из `targets` taxonomy.
3. `validate_patches.py`: кламп/снап/анти-тишина → экспорт каждого патча в `Patches/[TEST]_<concept>.json`.
4. Запустить синтезатор, прослушать каждый, пометить ✅ / ❌ / 🔧.
5. Ревью описаний: соответствуют ли звуку, разнообразны ли по стилям, нет ли цифр октавы/LFO в образных стилях.
6. По итогам — правки system-промпта/эвристик/param_reference (а не диапазонов архетипов).

**Критерии качества патчей:** каждая цель узнаваема; внутри категории слышимая вариативность; нет сломанных (тишина/клип/монотонный buzz).
**Критерии описаний:** по тексту понятен звук; 6 стилей различимы; нет противоречий параметрам; в #4/#5/#6 нет явных цифр.
