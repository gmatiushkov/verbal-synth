# VerbalSynth — План создания датасета (v2)

> Рабочий документ. v2 — пересмотр подхода: **Claude проектирует патчи и описания вместе**, а не описывает случайно сгенерированные.
> Связанные файлы: [PROJECT.md](PROJECT.md), [ML_IMPLEMENTATION.md](ML_IMPLEMENTATION.md),
> [ml/config/param_reference.json](ml/config/param_reference.json),
> [ml/config/sound_taxonomy.json](ml/config/sound_taxonomy.json),
> [ml/config/system_prompt.md](ml/config/system_prompt.md),
> [ml/config/wavetable_doc.json](ml/config/wavetable_doc.json).

---

## Цель

Датасет `(русский текст → 38 нормализованных float)` для обучения MiniLM+MLP.
Модель должна понимать технические, образные, референсные (инструменты) и эмоциональные описания.

---

## Главное изменение v1 → v2

**Было (v1):** скрипт случайно генерирует патчи в рамках диапазонов архетипа → Claude по сухим числам пишет описания.

**Стало (v2):** **Claude сразу проектирует конкретный звук** — придумывает задумку, выставляет 38 параметров под неё и пишет 6 описаний. Одним запросом он отдаёт патч + описания.

**Почему так лучше:**
- Случайные патчи в диапазонах получаются однообразными и безликими; по сухим числам Claude трудно придумать осмысленное описание.
- Когда Claude сам проектирует звук под идею («виолончель в низком регистре»), патч получается *интенциональным и связным*, а описание гарантированно ему соответствует.
- Разнообразие достигается прямым мандатом на вариативность, а не нарезкой случайных чисел.

**Следствие:** система архетипов с жёсткими диапазонами параметров **упраздняется**. Вместо неё —
[sound_taxonomy.json](ml/config/sound_taxonomy.json): карта пространства звуков (категории + примеры целей + оси разнообразия) **без** диапазонов параметров.

**Что нужно дать Claude, чтобы он «слышал» синтезатор:**
1. Полную таблицу 38 параметров с маппингом `0..1 → реальные величины` — [param_reference.json](ml/config/param_reference.json) (учитывает лог-кривые: cutoff 20–18000 Гц, времена 0.5мс–10с и т.д.).
2. Перцептивную документацию банков (`perceptual_summary` + `position_map`) — уже готова в [wavetable_doc.json](ml/config/wavetable_doc.json) (Фаза 0).
3. Эвристики саунд-дизайна (как получить плак/пэд/бас/колокол/вибрато…).
4. Мандат на разнообразие + спектр целевых звуков (имитации инструментов, характеры).
5. Правила 6 стилей описаний + словарь дескрипторов.

Всё это собрано в [system_prompt.md](ml/config/system_prompt.md).

## Решения (актуальные)

| Вопрос | Решение v2 |
|--------|-----------|
| Направление генерации | **Описание+патч вместе**, проектирует Claude (раньше: параметры→описания) |
| Архетипы с диапазонами | **Упразднены** → sound_taxonomy.json (категории без диапазонов) |
| Описаний на патч | 6 стилей (functional, instrument, psychoacoustic, emotional, metaphor, telegraphic) |
| Дискретные параметры | Claude ставит точные значения из snap-наборов; цифрами в описаниях не упоминаются |
| Язык описаний | только русский |
| Источник «слуха» Claude | param_reference.json + wavetable_doc.json в system-промпте |
| Валидация патчей | кламп [0,1] + снап дискретных + анти-тишина/клиппинг (правила в param_reference.json) |
| Тариф API | **Claude Default (0.7×)** — 1M-контекст не нужен |
| Модель | **claude-opus-4-8** (качество проектирования); Sonnet 4.6 — дешёвый запас |
| Удешевление | prompt caching системного промпта + Batch API (−50%, если поддержан прокси) |
| Формат вывода | structured outputs (JSON-схема), без текста вне JSON |

## Конфигурация API

Прокси aiprimetech (Anthropic-совместимый SDK).

```python
import anthropic
client = anthropic.Anthropic(
    api_key="YOUR_AIPRIMETECH_KEY",     # вставить позже
    base_url="https://api.aiprimetech.io/v1",
)
MODEL = "claude-opus-4-8"
```

### Выбор тарифа (бюджет $25)

| Тариф | Коэф. | Контекст | Вывод |
|-------|------|----------|-------|
| **Default** ✅ | 0.7× | 200K | Оптимум цена/мощность |
| Max | 1.3× | 1M, 99.6% cache hit | Только если нужен 1M-контекст |

Берём **Default**: системный промпт ~10–14K токенов (param-таблица + банки + правила) свободно влезает в 200K; платить 1.3× за 1M-контекст незачем. Default почти вдвое дешевле Max.

### Как делать запросы

- **System-промпт = стабильный префикс** (синт-доки + банки + словари + схема) → один `cache_control: {type:"ephemeral"}` на последнем system-блоке. Читается из кэша во всех запросах (~0.1× стоимости).
- **User-сообщение = переменная часть:** «Сгенерируй N патчей категории X, цели: [...], варьируй по осям [...]».
- **structured outputs** (`output_config.format` с json_schema) — гарантирует парсимый JSON. Opus 4.8 поддерживает.
- **Adaptive thinking** включён (лучше проектирует), `effort:"medium"` для экономии; батчинг 6–8 патчей/запрос амортизирует «размышления».
- **Batch API** (`/v1/messages/batches`, −50%) для массовой генерации — не латентно-критично. Если прокс не поддерживает batch — работаем в обычном режиме (бюджет всё равно проходит).
- **Чекпойнты:** дописывать результат в JSONL построчно (append-only), уметь продолжить с места обрыва.

### Оценка стоимости (Opus 4.8, Default 0.7×)

Допущения: ~1000 выходных токенов на патч (вкл. долю thinking + JSON + 6 RU-описаний); система кэшируется.

| Сценарий | Патчей | Out-токены | Стоимость вывода | + Batch −50% |
|----------|--------|-----------|------------------|--------------|
| Тест | ~60 | ~60K | ~$1.1 | ~$0.5 |
| Полный v1 | ~800 | ~800K | ~$14 | ~$7 |

Вход/кэш-чтение системного промпта ≈ $0.5 суммарно (незначимо). Итог: **Opus Default + Batch ≈ $7–8** на полный датасет → укладывается в $25 с запасом на итерации/перегенерацию. Без Batch ≈ $14–15 — тоже проходит. Если нужно экономить ещё — часть категорий генерировать Sonnet 4.6.

## Фаза 1 — Тестовая выборка (проверка качества)

**Цель:** убедиться, что патчи звучат как заявлено и описания адекватны — по **всем типам**, а не 4 архетипам.

**Объём:** ~3–5 патчей × каждая из 4 категорий = ~16–20 патчей × 6 = ~120 примеров.
Внутри каждой категории просить разные цели (например для acoustic_emulation: гитара, виолончель, флейта, орган, колокол) — чтобы проверить разные банки/приёмы.

### Шаги
1. Написать `generate_dataset.py` (сборка system-промпта из конфигов; structured outputs; чекпойнты).
2. Запустить в режиме `--test`: по N патчей на категорию, цели берутся из `targets` taxonomy.
3. `validate_patches.py`: кламп/снап/анти-тишина → экспорт каждого патча в `Patches/[TEST]_<concept>.json`.
4. Запустить синтезатор, прослушать каждый, пометить ✅ / ❌ / 🔧.
5. Ревью описаний: соответствуют ли звуку, разнообразны ли по стилям, нет ли цифр октавы/LFO в образных стилях.
6. По итогам — правки system-промпта/эвристик/param_reference (а не диапазонов архетипов).

**Критерии качества патчей:** каждая цель узнаваема; внутри категории слышимая вариативность; нет сломанных (тишина/клип/монотонный buzz).
**Критерии описаний:** по тексту понятен звук; 6 стилей различимы; нет противоречий параметрам; в #4/#5/#6 нет явных цифр.

## Фаза 2 — Полный датасет

После подтверждения тест-выборки:
1. `generate_dataset.py --all` → ~800 патчей по долям taxonomy (предпочтительно Batch API).
2. `validate_patches.py --all` → отчёт по подозрительным (анти-тишина/клип), точечная перегенерация/правка.
3. Контроль: прослушать 8–10 случайных патчей каждой категории.
4. Развернуть в `dataset.jsonl` (~4800 строк), финальный ревью 50 случайных примеров.

## Фаза 3 — Обучение и интеграция

Без изменений относительно [ML_IMPLEMENTATION.md](ML_IMPLEMENTATION.md): MiniLM-L6 (заморожен) + MLP → 38 sigmoid; HuberLoss; экспорт ONNX; интеграция кнопки Generate в `MainComponent.cpp` через Python subprocess / ONNX.

## Механизм расширения

Добавить новый тип звука:
1. Дописать цели/категорию в `sound_taxonomy.json` (диапазоны не нужны).
2. `generate_dataset.py --category new_name --count N`.
3. Прослушать выборку, поправить эвристики в system_prompt при необходимости.
4. Новые строки дописываются в `dataset.jsonl` (append-only); старое не трогается.
